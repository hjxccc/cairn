# 001 —— 退款处理用幂等键而非分布式锁

- 状态：现行
- 日期：03-15
- 来源：`tasks/03-15-refund-double-credit/`（决策在哪次任务里诞生）

## 背景
并发回调投递导致重复退款入账。我们需要退款执行的恰好一次**效果**。

## 候选
- A. 每退款意图一个幂等键，DB 唯一约束兜底（✅ 选定）
- B. 退款执行外套分布式锁（Redis）（❌ 否：锁 TTL 对不上慢网关调用——过早过期照样重复入账，过晚过期又堵队列；还给资金链路凭空引入 Redis 依赖）
- C. 只在回调消费端去重（❌ 否：覆盖不了**自家重放任务**的重试，而那才是真正的第二个写入方）

## 理由
资金链路上数据库本就是事实源；唯一约束是唯一一个"不管写入方是谁都成立"的护栏。

## 后果
退款表永久带一个意图键列；所有新写入方都必须提供它（用 NOT NULL + 代码评审清单一行强制）。
